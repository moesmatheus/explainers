# Preferences

## Process rules (top of file — read first)
- [2026-04-25] **NEVER do an explainer build in one go.** User-stated rule, direct feedback: writing a 2,500+ line file in a single Write call sometimes crashes Claude (token limits, context drift, parser ambiguity in long JSX). The mandated workflow is the **stage-by-stage build**: (Stage A) scaffolding only — primitives + Hero + SectionNav + GLOSS + stub default export; verify it loads. (Stage B) register in `src/explainers.js` + verify scaffolding renders. (Stages C-I) add 1-2 cards per stage, run `npx vite build` after each Edit batch, eval the page after each stage to confirm card IDs render and KaTeX has zero errors. (Stage J) full QA detector sweep. (Stage K) update PREFERENCES.md. This applies to ALL future explainer builds regardless of perceived simplicity. Don't try to be clever and batch — the cost of one crashed turn is much higher than the cost of 10 small turns. _user feedback after the C build_

- [2026-05-21] **Estimator-scorecard anchor for a methods-survey explainer** — for any "here are N ways to estimate X" explainer, the anchor is a leaderboard: run every method on ONE dataset that has a known ground truth, render each as a row with a point estimate, a CI whisker, and the truth as a dashed vertical line. Color emerald if the CI/estimate lands near truth, rose if not. Predict-then-reveal: the reader commits a guess for "the best estimate" before the board unlocks. The killer property is a dataset with a *known answer* — the LaLonde/NSW job-training data works because a randomized experiment gives the true ATT (~+$1,800), so observational estimators can be objectively scored. Let some methods honestly FAIL on the board (diff-in-diff overshot because parallel trends fails on NSW) — a scoreboard where everything succeeds teaches less than one where method choice visibly matters. _causal-inference/Anchor_
- [2026-05-21] **Shared interactive-DAG primitive (`CausalDAG`)** — one SVG component taking `nodes` + `edges` arrays, reused across 4 cards (confounding, do-operator, back-door/collider, mediation). - [2026-05-21] **Single latent "confounder index" drives BOTH selection and outcome** — for an explainer where many observational estimators must all recover a known effect, generate the synthetic data so ONE index (a linear combo of covariates) drives both the treatment-assignment propensity AND the baseline outcome. Then unconfoundedness genuinely holds and the outcome is linear in the covariates, so regression / matching / IPW / double-ML all converge on the truth. Building selection and outcome from independently-chosen covariate weightings does NOT work — the propensity stops being a sufficient statistic for the outcome and every estimator inherits a different bias. The data-generating process, not the card UI, is where a methods explainer lives or dies. _causal-inference/buildNSW_

## What worked well
- [2026-05-21] **Define all content cards as named stub components in the scaffolding stage; the App export never changes.** Stage A wrote 19 named components (`LadderCard`, `PotentialCard`, …) each rendering a shared `<StubCard>`, and the default export listed all 19 from the start. Later stages replace each component's *body* in place. This sidesteps the data-centers-br gotcha entirely — there is never a dangling `<StubCard>` reference because the App always references real, defined component names. Zero render-wipe incidents across a 19-card build. _causal-inference build_
- [2026-05-21] **Expose the synthetic dataset on `window` during the build to debug estimator calibration, then remove it in QA.** A methods explainer's hard part is making the live-computed estimates actually land on the known truth. Adding `if (typeof window !== 'undefined') window.__NSW = NSW` let me re-run estimator math in `preview_eval` with the real data — sweeping ridge λ, caliper, k-NN variants — without rebuilding each time. Found the noise-concentration bug, the propensity-calibration bug, and the e-vs-logit matching bug this way. Delete the hook in the QA stage. _causal-inference build_
- [2026-05-21] **Verify every estimator's numeric output via `preview_eval` after each stage, not just "does it render".** For a methods explainer the cards can render perfectly while the estimate is silently wrong (matching returned +$3,568 when truth was +$1,800). Each stage's eval pulled the actual Stat-tile text and checked it against the known ATT. Catching a miscalibrated estimator at its own stage is far cheaper than discovering it at the anchor scorecard. _causal-inference build_
- [2026-05-10] **Stage-by-stage with 2 cards per stage on a 20-card explainer** worked smoothly. Per-stage cycle: (a) one Edit appending two card components, (b) one Edit updating the main render to slice past the new pair, (c) `npx vite build` to verify parse, (d) `preview_eval` to verify section IDs render and katexErrors=0. ## Gotchas (don't repeat)
- [2026-05-21] **An apostrophe inside a single-quoted KaTeX/JS string silently terminates the string.** Wrote `<Block>{'… X{=}x',Y{=}y') …'}</Block>` — the `x'` ended the string, and esbuild threw `Expected "}" but found "{"` pointing at the Block line. Fix: use double-quoted JS strings for any TeX containing apostrophes (`{"… x',y' …"}`), or `&apos;`-free phrasing. Same class as the deep-uncertainty backslash-escape gotcha — any time a TeX string has a `'`, switch the JS quote style. _causal-inference/Ladder_
- [2026-05-21] **`.toLocaleString()` with no locale arg uses the browser's default locale — which may format `1809` as `1.809`.** Computed-currency strings rendered with a `.` thousands separator on a non-US-locale browser, so "+$1,809" displayed as "+$1.809". Always pass `'en-US'` explicitly: `n.toLocaleString('en-US')`. The repo's `fmtK` helper already does this; the bug was in ad-hoc `${Math.round(x).toLocaleString()}` calls added later. Grep `toLocaleString()` (empty parens) before QA. _causal-inference/Anchor, Sensitivity_
- [2026-05-21] **`cd` inside a Bash tool call persists into the next call and breaks `npx vite build`.** Ran `cd causal-inference && sed -i '' …` to bulk-edit; the next `npx vite build` then failed with `Could not resolve entry module "index.html"` because the shell was still in the subdirectory. The Bash tool keeps working directory across calls. Either `cd` back explicitly, or run one-off commands with an absolute path and no `cd`. Also: editing a file with `sed` outside the Edit tool makes the harness lose its file-state tracking — the next `Edit` then errors "File has not been read yet" and needs a fresh `Read`. _causal-inference build_
- [2026-05-21] **Nearest-neighbour matching WITH replacement under thin overlap concentrates on a handful of controls, so outcome noise does not average out — the estimate is biased, not just noisy.** Propensity matching returned +$3,568 vs a +$1,800 truth because ~15 distinct control units served as matches for all treated units, and that tiny set's idiosyncratic noise did not cancel. Three fixes together cured it: (a) far more control units so the overlap region is densely populated, (b) lower outcome noise σ, (c) k-NN (k≈15) averaging instead of 1-NN. Also: **match on the propensity LOGIT, not the squashed score `e`** — equal `e`-distance can mean very different logit/covariate distance near the extremes (Jensen). _causal-inference/Matching_
- [2026-05-21] **IPW needs a *calibrated* propensity (a fitted logistic), not a hand-set monotone score; and it fails quietly under poor overlap.** A hand-tuned `e(X)` that merely ranks units correctly is fine for matching (rank-only) but wrong for IPW, whose `e/(1-e)` weights depend on the actual probability — using an uncalibrated score gave IPW −$5,401 vs +$1,800 truth. Fix: fit a real logistic regression of D on covariates (gradient descent, light ridge, quadratic terms because earnings had different variances per group → the true propensity is quadratic). Even calibrated, IPW on poor-overlap data gives a correct point estimate on a tiny effective sample (ESS ≈16 of 2,500) — present that honestly with a weight-concentration curve rather than forcing a trim slider (trimming 1% of weights swung the estimate from +$1,900 to −$400). _causal-inference/IPW_
- [2026-05-21] **For double ML, use the doubly-robust AIPW form with propensity clipping — the partialling-out (PLR) estimator is biased when both nuisance models are imperfect.** Residual-on-residual partialling-out gave +$2,374 (truth +$1,800) because the linear outcome model could not capture the `τ·e(X)` term and the propensity was approximate — the product of the two nuisance errors is a real second-order bias. AIPW with a separate outcome model fit on controls only (correctly specified) plus a clipped-propensity correction landed at +$2,000. Clip `e` to ~[0.1, 0.9] so the IPW-correction term does not inherit the ESS-collapse problem. _causal-inference/DoubleML_
- [2026-05-10] **Dead-variable expressions still execute and can throw at runtime even if their result is unused.** Wrote `const gaussScale = ... * Math.max(...arr * X) ...` where `arr * X` evaluates to NaN, then spread on NaN throws "argument is not iterable" on every render. Build passed (parse-clean), but the page silently caught the error in React's wrapper and the eval reported `sections: 0`. Diagnostic: `preview_console_logs level:error` showed the throwing component name (`NonGaussCard`) immediately. Don't write expressions that won't be used; if you do, make sure they don't throw — or just delete them. _retail-quant/NonGauss_
- [2026-05-10] **Numeric-format detector triggers on raw computed dollar values too**, not just on hardcoded literals. The anchor card had `${(netRet * 250000).toFixed(0)}` which evaluates to `11625` (5 digits) at render time and gets flagged. Fix: use `.toLocaleString()` after `Math.round()` for any computed dollar value going into JSX text. Safer pattern overall when displaying derived currency. _retail-quant/Anchor_
- [2026-04-29] **Tailwind `truncate` on a `grid-cols-[120px_…]` cell silently clips capex/value strings.** Symptom in detector 6 (HTML truncation): 6 hits in the Financeiro card summary table, all of them currency strings like "R$ 6,2B SP + R$ 3B RS (fase 1)" landing in a 120 px column. Fix: collapse the multi-column row to a stacked content layout — instead of `grid-cols-[120px_1fr_120px_90px]` with 4 truncated cells, use `grid-cols-[100px_1fr_90px]` and inside the middle cell stack the 3 metrics vertically with `space-y-0.5` and a tiny uppercase eyebrow per line ("rec", "ebitda", "capex"). Wraps naturally, no clip, reads better. Lesson: when data is genuinely long-form (not a single number), don't fight it with `truncate` + small fixed widths; let it wrap. _data-centers-br/Financeiro_
- [2026-04-29] **Inserting a new card into the SECTIONS array shifts every downstream `index={N}` prop.** Adding `Financeiro` between `Capital` (was 6) and `Estrategia` (was 7) means the eyebrow numbers on Estrategia, Clientes, Diferenciais, M&A, Posicionamento, Fontes, and Trails all need to bump by 1. Easy to miss because the page still renders fine — the badges just look stale. Always grep `index=\{[0-9]+\}` after adding a new section card and audit sequentially. The SECTIONS array uses position-based indexing for the side-nav so it auto-corrects, but the eyebrow numbers on the Card components are hard-coded. _data-centers-br Stage 2_
- [2026-04-29] **Deleting a stub component while migrating its references — silent React tree wipe.** Symptom: `npx vite build` passes, body renders 0 chars at runtime, `[error] The above error occurred in <Component>` in console with no prior message. Cause: replaced `StubCard` definition with new card components in one Edit, but kept 11 `<StubCard>` references for the cards not yet migrated. ReferenceError at module evaluation, but neither vite-build nor browser console surfaced the underlying error name — only the React error boundary message. Diagnostic that worked: `grep -n 'StubCard' <file>` showed 11 references with no matching definition. Lesson: when migrating from stubs, always grep the symbol in question after the Edit; build success ≠ render success when the missing symbol is referenced inside JSX (transpiles to identifier, defers binding to render time). _data-centers-br Stage C_
- [2026-04-29] **Forward reference to a function declared after a `const` array — `Table2Like` ReferenceError pattern.** Even though function declarations are hoisted, that hoisting is to the top of the *function/module* scope. If you're reading a `const SECTIONS = [{ icon: Table2Like, ... }]` BEFORE the `function Table2Like(...) {}` declaration in the same module, the function name resolves but isn't always wired up correctly with React Refresh / Vite's transformations — the icon ends up undefined at render. Fix: either declare the function first, or just use the underlying icon directly (here `Database`). Rule of thumb in this repo: don't write tiny aliasing function components — use the real icon. _data-centers-br Stage A_

- [2026-04-25] **Backslash-quote escaping inside JSX attribute strings BREAKS the parser, even though backslash-apostrophe is "tolerated" elsewhere.** Writing `because="Taleb\'s point is that the assumption \\\"finite variance\\\" fails"` triggers `Expected ">" but found ","` from esbuild — the doubled `\\\"` ends the attribute prematurely. Two safe forms: (i) just write the apostrophe directly (`Taleb's point...`); (ii) drop the inner quote and use em-dash phrasing (`the assumption — finite variance — fails`); (iii) use a curly-brace child: `because={`Taleb's point...`}`. Don't escape with `\\\"`; never works. The build error is loud (HMR reports failed reload) but eval at the same moment can still return previous-render results, masking the issue — always run `npx vite build` after a heavy edit to catch parse errors immediately. _deep-uncertainty/FatTails — first failed reload in this codebase_
- [2026-04-25] **Edit success ≠ HMR success.** When Edit returns "successfully", the file IS updated, but Vite may fail to hot-reload it due to a syntax error introduced by the edit. Running `eval` against the page returns whatever the previously-loaded module is rendering. Symptom: cards array is `[]` despite the new code being on disk. Diagnostic: check `[vite] Failed to reload` in console logs OR run `npx vite build` to surface the parse error directly. Now part of the heavy-explainer workflow: build-check after each Stage. _deep-uncertainty build_
- [2026-04-25] **Section-nav `<a href="#section-id">` was kicking the hash router back to Home on every in-page anchor click.** Root cause: `App.jsx` does `explainers.find(e => e.slug === hash)` and falls back to `<Home/>` if no match — so clicking a side-nav link with `href="#calibration"` set the hash to `calibration`, no slug matched, and the user landed on Home. Affected all 11 explainers since they share the same hash-router. Single-file fix: in `App.jsx`, track `lastSlugRef.current` of the last-mounted explainer, and when `hash !== ''` and matches no slug, fall back to that ref instead of Home. Preserves URL-as-deep-link for slugs (`#superforecasting` works) AND in-page scroll-to-section (browser scrolls to `#calibration`, router stays on the explainer). _src/App.jsx_
- [2026-04-25] **`motion.circle` / `motion.text` animating SVG attributes (cx, x, etc.) warns "from undefined to N" even with explicit `initial`** in framer-motion v11. The fix that actually silenced it: drop motion entirely on the SVG element and use a plain `<circle cx={...} style={{ transition: 'cx 0.25s ease' }} />`. The CSS transition does the same job and React just diffs the attribute. Don't fight motion on bare SVG attributes — use motion for `style`/`transform`/CSS-numeric props, plain SVG with CSS transitions for `cx`/`x`/`r`/etc. _superforecasting/Aggregate_
- [2026-04-25] **End-of-line legend labels in a chart's SVG always overflow the viewBox at narrow viewports.** Pattern: `<text x={sx(maxX)+5}>label</text>` puts text past the right edge whenever `sx(maxX) + label_width > W-P`. Caught by the layout detector — superforecaster / regular forecaster / always-50% labels overflowed by 33–62 px. The reliable fix: lift the legend out of the SVG and render it above the chart as an HTML chip row (`<span class="inline-flex"><span class="w-3 h-0.5 bg-emerald-300" />superforecaster</span>`). Cheaper, narrower-viewport-safe, more consistent typography. _superforecasting/Horizons_
- [2026-04-25] **`whileInView` opacity transitions stall at intermediate values on very tall pages (~24k px, 12+ heavy cards).** Even the documented "scroll-to-top → wait → scroll-to-target → wait 1.5s" sequence sometimes lands a card at opacity ~0.05 instead of 1, because IntersectionObserver fires but the transition doesn't complete before the screenshot. For QA, prefer DOM-level eval verification over screenshot inspection. If a screenshot is required, run `document.querySelectorAll('section[id]').forEach(el => { el.style.opacity='1'; el.style.transform='none'; })` AFTER scroll, then screenshot — but note this still occasionally fails on heavy pages. _superforecasting QA run_
- [2026-04-25] **Tailwind dynamic class strings via `chipPalette[color].split(' ')[N]`** — using array-indexed splits of a literal class string DOES work because the literal is in the source, but is fragile: any palette refactor that reorders the words breaks every consumer. Better: structure the palette as an object `{ bg, text, border }` from the start, and pass `chipPalette[color].border` etc. Less clever, much easier to refactor. _superforecasting/Operationalize, ReferenceClasses_
- [2026-04-25] **Edit tool can return "successfully" when the edit silently didn't apply** — likely a whitespace/escape mismatch in `old_string`. Always verify with grep or fresh Read after a meaningful edit, especially on files you haven't touched recently. Don't trust the success response alone. _superforecasting PREFERENCES update_
- [2026-04-25] **In-SVG axis titles fight digit labels at the viewBox edge.** Putting `<text>` axis titles ("home goals →", "← away goals", "CV ½-width") inside an SVG near its borders almost always overlaps with the tick-digit labels or with the SVG bounding box itself. The detector flagged 3 overlaps + 1 left-overflow on the first pass. Fix: lift the axis titles OUT of the SVG entirely. Use a CSS grid wrapping the SVG with two cells: a vertical-orientation div on the side (`writingMode: vertical-rl; transform: rotate(180deg)` for a "↑ away goals" caption) plus a centered HTML caption below for the x-axis. Cleaner typography, no overlap, narrower-viewport-safe. _forecasters-craft/Distributional, Synth_
- [2026-04-25] **Side-nav `label` strings with `&` — pick one strategy and apply it consistently in both the desktop and mobile nav blocks.** Two forms work: (a) plain string `"Drift & online"` rendered as `{s.label}` (since the string literal contains a real `&`, React renders it correctly); (b) entity-encoded string `"Drift &amp; online"` rendered via `dangerouslySetInnerHTML={{ __html: s.label }}`. Don't mix — if one nav uses (a) and the other uses (b) you get visible literal `&amp;` text on one and the real ampersand on the other. _forecasters-craft/SectionNav_
- [2026-04-24] **JSX parses bare `<` as an element opener, even inside text nodes.** Writing `SaaS B2B < R$ 78M opta por Presumido` inside a `<p>` breaks compilation with "Identifier directly after number" pointing at the `<`. Fixes: use `&lt;`, replace with natural-language phrasing ("até R$ 78M", "menor que"), or wrap the comparison in `{}` as a string literal. This is a common Portuguese-language pitfall because "X menor que Y" translates mentally to `X < Y` and your fingers follow. _tributacao-brasil/PisCofinsCard_
- [2026-04-24] **Tailwind JIT needs literal class names, not template strings.** Writing `` className={`bg-${color}-500/20`} `` never compiles `bg-emerald-500/20` into the final CSS because Tailwind scans source for literals. Two workarounds that work: (i) pre-resolved lookup object `const palette = { emerald: { bg: 'bg-emerald-500/20', ... } }` passed to children; (ii) let the parent pass the full class string as a prop (`bar="bg-emerald-500/70"`). Trying to be clever with `.replace('text-', 'bg-')` at runtime on a literal class also fails — Tailwind still needs to see the literal at <i>build</i> time. _tributacao-brasil/MapaFederativo, Panorama_
- [2026-04-24] **framer-motion's `whileInView` makes programmatic scrolling + instant screenshot useless.** Cards sit at opacity 0 until the IntersectionObserver fires. When you `scrollIntoView` in a preview-eval IIFE, screenshot immediately after, you capture a black screen because the transition hasn't triggered. Fix: scroll to top first, wait ~400ms, scroll to target, wait ~1.5s — that gives the observer two state changes to latch. Otherwise the viewport's occupancy never changes and the card stays invisible. _tributacao-brasil QA run_
- [2026-04-24] **Preview MCP viewport defaults tiny and resets on reload.** The default preview size is often 240×716 until `preview_resize` is called, and sometimes reload reverts it. Always resize up front (desktop or explicit 1280×800) and re-resize after any navigation/reload. Otherwise QA detectors report hundreds of false-positive overflows that are just viewport narrowness. _tributacao-brasil QA run_
- [2026-04-19] **`position: fixed` tooltips get rebased by `transform` on an ancestor.** framer-motion `<motion.div>` wrappers (used inside `Card`) create a new containing block, so a naive `fixed` floating tooltip renders hundreds of pixels off-cursor and can clip on the right side of a card. Fix: portal the tooltip to `document.body` via `createPortal`. Also: measure the tooltip's actual `offsetWidth` in `useEffect` and flip it to the left of the cursor when `x + gap + width > vw - margin`. Pattern lives in the `FloatingTip` primitive in `world-economy/WorldEconomyExplainer.jsx`. _world-economy_
- [2026-04-17] **KaTeX macros and `#`**: inside a macro body, `#` starts a parameter ref (`#1`). Hex colors like `#c4b5fd` MUST be doubled to `##c4b5fd`, otherwise KaTeX throws "Not a valid argument number" and the equation falls back to raw red TeX. _quantum-mechanics_
